## 1. 개요

### 1.1 목적

본 SRS는 v2 데모 서비스가 제공해야 할 모든 기능과 품질 기준을 명시하여, 개발팀과 검증팀 간의 공통 이해를 확보한다.

### 1.2 범위

- 프론트엔드(React)의 사용자 대화형 기능
- 백엔드(Express)의 API 및 AI 중계 기능
- Python 빌드 스크립트의 HWPX 문서 생성 기능
- 자동화 인프라(ADR, skills, hooks, tools)

### 1.3 용어 정의

| 용어 | 정의 |
|------|------|
| HWPX | 한컴오피스의 XML 기반 개방 문서 포맷 (ZIP + XML) |
| HWP | 한컴오피스의 기본 이진 문서 포맷 |
| WASM | WebAssembly. 브라우저 내에서 네이티브 속도로 실행되는 바이너리 포맷 |
| rhwp | `@rhwp/core` 패키지. HWP/HWPX 파싱/렌더링 WASM 라이브러리 |
| Provider | AI 모델 제공자 (Anthropic, OpenAI, Kimi, xAI) |
| ARIA | Accessible Rich Internet Applications. 접근성 속성 표준 |
| ADR | Architecture Decision Record. 아키텍처 결정 기록 |

## 2. 시스템 기능 요구사항

### 2.1 파일 업로드 및 파싱 (FR-UPLOAD)

| ID | 요구사항 | 우선순위 |

| FR-UPLOAD-004 | 파싱 결과는 SVG 형태의 페이지 미리보기로 렌더링되어야 한다. | P0 |

## 3. 비기능 요구사항

### 3.1 성능 요구사항 (NFR-PERF)

| ID | 요구사항 | 목표치 |
|----|----------|--------|
| NFR-PERF-001 | WASM 파싱 완료 (SVG 렌더링) | ≤ 3초 (10페이지 기준) |
| NFR-PERF-002 | AI 초안 생성 완료 | ≤ 15초 (표준 네트워크) |
| NFR-PERF-003 | HWPX 파일 생성 완료 | ≤ 5초 |
| NFR-PERF-004 | 전체 흐름 (업로드 → 다운로드) | ≤ 30초 |
| NFR-PERF-005 | 클라이언트 메모리 사용량 | ≤ 200MB |
| NFR-PERF-006 | 다중 페이지 텍스트 추출(2~3페이지) | ≤ 2초 (비동기 백그라운드) |

### 3.2 보안 요구사항 (NFR-SEC)

| ID | 요구사항 |
|----|----------|
| NFR-SEC-001 | 서버는 `127.0.0.1`에만 바인딩되어야 한다. |
| NFR-SEC-002 | API 키는 서버 메모리 또는 `.env`에만 저장되며, 클라이언트로 노출되지 않아야 한다. |
| NFR-SEC-003 | 업로드 파일은 생성 프로세스 종료 후 즉시 삭제되어야 한다. |
| NFR-SEC-004 | CORS는 설정된 `CLIENT_ORIGIN`에서만 허용해야 한다. |
| NFR-SEC-005 | 파일 Magic bytes(`PK\x03\x04`) 검증으로 위조 파일 업로드를 방지해야 한다. |

### 3.3 신뢰성 요구사항 (NFR-REL)

| ID | 요구사항 |
|----|----------|
| NFR-REL-001 | AI 응답 파싱 실패 시 최소 1회 자동 재시도해야 한다. |
| NFR-REL-002 | `cairosvg` 미설치 환경에서도 다이어그램 없이 HWPX 생성이 가능해야 한다. |
| NFR-REL-003 | Python 빌드 스크립트 예외 발생 시에도 임시 파일이 남지 않도록 cleanup해야 한다. (`finally` 블록) |
| NFR-REL-004 | WASM 파싱 실패 시에도 앱이 완전히 죽지 않고 오류 메시지를 표시해야 한다. |

### 3.4 사용성 요구사항 (NFR-UX)

| ID | 요구사항 |
|----|----------|
| NFR-UX-001 | 별도 Docker 없이 `npm install && npm run dev`로 즉시 실행 가능해야 한다. |
| NFR-UX-002 | AI Provider 설정 없이도 기본 동작(키 미설정 시 안내 메시지)이 가능해야 한다. |
| NFR-UX-003 | 각 단계(파싱, 생성, 납품)의 진행 상태를 사용자에게 텍스트로 표시해야 한다. |
| NFR-UX-004 | 드래그 앤 드롭 시 시각적 피드백(색상 변경, 텍스트 변경)을 제공해야 한다. |

### 3.5 접근성 요구사항 (NFR-A11Y)

| ID | 요구사항 |
|----|----------|
| NFR-A11Y-001 | 키보드만으로 모든 주요 흐름(업로드, 생성, 다운로드)을 완료할 수 있어야 한다. |

## 4. 인터페이스 요구사항

### 4.1 사용자 인터페이스

- **파일 업로드**: 드래그 앤 드롭 존 + `<input type="file" accept=".hwp,.hwpx">`
- **문서 유형 선택**: `<select>` (report, proposal, minutes, gonmun, base)
- **텍스트 입력**: 회사명, 제목, 목표, 메모 (`<input>`, `<textarea>`)
- **미리보기 패널**: SVG 렌더링 영역 + 메타 정보 그리드
- **설정 패널**: Provider 선택, API 키 입력, 테스트 버튼

### 4.2 하드웨어/소프트웨어 인터페이스

| 구성요소 | 인터페이스 |
|----------|-----------|
| 브라우저 ↔ WASM | `@rhwp/core` HwpDocument API |
| 클라이언트 ↔ 서버 | HTTP REST API (JSON + multipart/form-data) |
| 서버 ↔ AI Provider | 각 Provider의 공식 SDK (HTTP) |
| 서버 ↔ Python | `child_process.spawn` (`python3 scripts/build_hwpx.py`) |
| Python ↔ HWPX | ZIP 압축 해제/생성, lxml XML 조작 |

## 5. 제약사항

| 제약 ID | 내용 |
|---------|------|
| CON-001 | 출력 포맷은 HWPX만 지원하며, 레거시 HWP 출력은 지원하지 않는다. |
| CON-002 | 한국어 문서만 지원한다. (프롬프트, UI 모두 한국어 기준) |
| CON-003 | 브라우저는 WASM를 지원하는 최신 버전(Chrome 90+, Safari 15+, Firefox 88+)을 요구한다. |
| CON-004 | Python 3.9+ 및 `cairosvg`(선택)이 시스템에 설치되어 있어야 한다. |
| CON-005 | 동시 다중 사용자 환경을 고려하지 않는다. (단일 사용자 localhost 데모) |
