## 1. 개요

본 문서는 v2 데모 서비스의 일상적인 운영, 모니터링, 문제 해결 방법을 안내한다.

## 2. 일상 운영 절차

### 2.1 서버 시작

```bash
cd /path/to/calendar_app/v2
npm run dev
```

출력 예시:
```
[v2 server] v2 server listening on http://127.0.0.1:8790
[v2 client] VITE v5.4.x  ready in 312 ms
[v2 client] ➜  Local:   http://localhost:5190/
```

### 2.2 버전 A 실행 (자동 로그인 오버레이)

```bash
cd /path/to/calendar_app/v2
npm run dev:auto
```

출력 예시:
```
[v2 server] v2 server listening on http://127.0.0.1:8791
[v2 client] VITE v5.4.x  ready in 312 ms
[v2 client] ➜  Local:   http://localhost:5191/
```

> 버전 B(5190/8790)와 A(5191/8791)를 동시에 실행할 수 있다.

### 2.3 서버 종료

터미널에서 `Ctrl + C`를 두 번 눌러 client와 server 모두 종료한다.

### 2.4 생성 파일 정리

생성된 `.hwpx` 파일은 `v2/generated/`에 누적된다. 주 1회 정리를 권장한다.

```bash
# 생성 파일 전체 삭제 (주의: 복구 불가)
rm v2/generated/*.hwpx
rm v2/generated/*-sections.json
```

## 4. 인증 및 API 키 관리

### 4.1 Google OAuth 설정

`v2/server/.env`에 Google Cloud Console에서 발급받은 OAuth 2.0 클라이언트 ID를 설정한다.

```bash
# server/.env
GOOGLE_CLIENT_ID=xxxx.apps.googleusercontent.com
GOOGLE_CLIENT_SECRET=xxxx
OAUTH_REDIRECT_URI=http://localhost:8790/auth/google/callback
```

> **참고**: 미설정 시 Mock 로그인 폼으로 자동 폭백된다. 개발 및 테스트에는 Mock 로그인으로 충분하다.

### 4.2 세션 관리

- 세션은 **메모리 기반**으로 저장되며, 서버 재시작 시 초기화된다.
- `v2_session` 쿠키는 `httpOnly`, `SameSite=Lax`, 24h TTL로 설정된다.
- 로그아웃 시 `POST /api/logout`으로 세션이 즉시 삭제된다.

### 4.3 API 키 관리

### 4.1 키 갱신

1. `v2/server/.env` 파일 편집
2. 해당 Provider의 API 키 값 변경
3. 서버 재시작 (`npm run dev`)

### 4.2 키 유출 의심 시

1. 즉시 해당 Provider 대시보드에서 키 폐기
2. `.env` 파일에서 유출된 키 삭제
3. 새 키 발급 후 `.env`에 반영
4. 서버 재시작

## 5. 로그 확인

### 5.1 클라이언트 로그

- 브라우저 개발자 도구 (F12) → Console 탭
- WASM 파싱 상태, AI 호출 결과, 렌더링 에러 확인

### 5.2 서버 로그

터미널에서 `npm run dev` 실행 중인 창의 stdout/stderr를 확인한다.

주요 로그 패턴:

| 로그 | 의미 | 조치 |
|------|------|------|
| `v2 server listening on ...` | 서버 정상 기동 | 불필요 |
| `cairosvg unavailable` | 다이어그램 변환 불가 | `brew install cairo` 또는 무시 |
| `AI 호출 실패: timeout` | Provider 응답 지연 | 네트워크 확인, 잠시 후 재시도 |
| `sections JSON parse failed` | 클라이언트 전송 데이터 오류 | 개발자에게 문의 |

## 6. 문제 해결 (Troubleshooting)

### 6.1 WASM 파싱이 안 됨

**증상**: 파일 업로드 후 "파서를 로드할 수 없습니다" 또는 빈 화면

**원인 및 해결**:
1. 브라우저가 WASM를 지원하지 않음 → Chrome 최신 버전 사용
2. 보안 소프트웨어가 WASM 차단 → 예외 설정 추가
3. `node_modules/@rhwp/core` 누락 → `npm install` 재실행

### 6.2 AI 초안 생성 실패

**증상**: "AI 서버 연결에 실패했습니다" 또는 502 에러

**원인 및 해결**:
1. API 키 미설정 → 설정 패널 또는 `.env` 확인
2. API 키 잘못됨 → Provider 대시보드에서 키 확인
3. Provider 서버 다운 → 상태 페이지 확인, 다른 Provider 전환
4. 네트워크 불안정 → VPN/방화벽 확인

### 6.3 HWPX 생성 실패

**증상**: "HWPX 생성에 실패했습니다" 또는 500 에러

**원인 및 해결**:
1. Python 3 미설치 → `python3 --version` 확인
2. 템플릿 파일 누락 → `templates/gonmun.hwpx` 존재 확인
3. 디스크 공간 부족 → `df -h` 확인
4. 업로드 파일 손상 → 다른 파일로 테스트

### 6.4 생성된 HWPX가 한컴에서 깨짐

**증상**: 한컴오피스에서 열었을 때 글자 겹침, 서식 어긋남

**원인 및 해결**:
1. 원본 양식의 특수 스타일 미지원 → 기본 템플릿(`gonmun.hwpx`) 사용 테스트
2. `_normalize_paragraph`의 linesegarray 리셋 문제 → 개발자에게 보고
3. 폰트 미설치 → 한컴에서 대체 폰트로 자동 표시됨 (정상 동작)

### 6.5 포트 충돌

**증상**: `EADDRINUSE` 에러

**해결**:
```bash
# 사용 중인 프로세스 확인 및 종료
lsof -i :5190
lsof -i :8790
lsof -i :5191
lsof -i :8791
kill -9 <PID>
```

### 6.7 로그인/세션 문제

**증상**: 로그인 버튼 클릭 후 아무 반응 없음 또는 `/api/me`가 항상 `authenticated: false`

**원인 및 해결**:
1. 팝업 차단 → 브라우저 팝업 차단 해제
2. CORS credentials 미설정 → `vite.config.js`와 `server/index.js`의 `credentials: true` 확인
3. `cookie-parser` 누락 → `npm install cookie-parser` 후 서버 재시작
4. 서버 재시작으로 세션 초기화 → 정상 동작, 다시 로그인 필요

### 6.6 파일 해제 후 재업로드 불가

**증상**: 파일 해제(×) 후 같은 파일을 다시 선택필 수 없음

**원인**: `<input>`의 `value`가 초기화되지 않음

**해결**: Uploader에서 `event.target.value = ''`로 초기화 처리됨 (이미 구현됨). 재현 시 브라우저 캐시 초기화.
